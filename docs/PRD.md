# Faceology PRD (제품 요구사항 문서)

- 문서 버전: v1.0
- 작성일: 2026-02-14
- 문서 목적: 카메라로 사용자 얼굴을 촬영하고, 엔터테인먼트 목적의 관상 해석 결과를 제공하는 모바일/웹 앱의 제품 요구사항 정의

## 1) 제품 개요

### 1.1 제품명
- 가칭: Faceology

### 1.2 문제 정의
- 사용자는 자신의 얼굴 사진을 기반으로 재미 요소가 있는 해석형 콘텐츠를 빠르게 경험하고 공유하고 싶다.
- 기존 관상 콘텐츠는 텍스트 위주이거나 입력 과정이 번거롭고, 개인정보 처리 기준이 불명확한 경우가 많다.

### 1.3 제품 목표
- 10초 이내 촬영/분석/결과 확인이 가능한 간편한 사용자 경험 제공
- 재방문을 유도하는 결과 카드 및 공유 기능 제공
- 얼굴 이미지 처리에 대한 명확한 동의/보안/삭제 정책 제공

### 1.4 제품 원칙
- 본 서비스는 **엔터테인먼트 목적**으로 제공한다.
- 의학/법률/채용/신용 등 고위험 의사결정에 사용하지 않는다.
- 결과는 결정적 사실이 아닌 확률적/해석형 콘텐츠로 제공한다.

## 2) 타겟 사용자

### 2.1 핵심 타겟
- 18~39세 모바일 친화 사용자
- 가벼운 자기해석/심리테스트/소셜 공유 콘텐츠 선호층

### 2.2 사용자 니즈
- 로그인 없이 빠르게 체험하고 싶다.
- 결과를 이미지 카드로 저장/공유하고 싶다.
- 사진과 개인정보가 안전하게 처리되는지 알고 싶다.

## 3) 범위 정의

### 3.1 MVP 포함 범위
- 카메라 권한 요청 및 얼굴 촬영
- 사진 업로드(갤러리 선택)
- 얼굴 탐지(단일 얼굴 기준) 및 품질 체크(밝기/정면 여부)
- 관상 해석 결과 생성(성향, 대인관계, 일/금전, 오늘의 한 줄)
- 결과 카드 생성 및 공유(이미지 저장, 링크 공유)
- 동의 화면(개인정보/생체정보 처리 안내)
- 사용자 요청 기반 얼굴 이미지 즉시 삭제 기능

### 3.2 MVP 제외 범위
- 실시간 영상 분석
- 다중 인물 동시 분석
- 결제/구독
- 전문가 1:1 상담
- 채팅형 상담 에이전트

## 4) 사용자 시나리오

### 4.1 주요 사용자 흐름
1. 앱 실행
2. 서비스 안내 및 동의 확인
3. 카메라 촬영 또는 사진 업로드
4. 얼굴 인식 및 품질 검증
5. 분석 진행(로딩 1~3초)
6. 결과 화면 확인
7. 결과 카드 저장/공유
8. 재촬영 또는 종료

### 4.2 예외 흐름
- 얼굴 미검출: “얼굴이 화면 중앙에 오도록 다시 촬영해 주세요” 안내
- 다중 얼굴 감지: “한 명만 촬영해 주세요” 안내
- 저화질: “조명을 밝게 하고 정면으로 촬영해 주세요” 안내
- 동의 미체크: 서비스 이용 불가 처리

## 5) 기능 요구사항

### 5.1 촬영/업로드
- FR-001: 카메라 권한 요청 및 거부 시 재요청 가이드 제공
- FR-002: 전면/후면 카메라 전환 지원
- FR-003: 갤러리 업로드 지원(JPG/PNG, 최대 10MB)

### 5.2 얼굴 처리
- FR-004: 단일 얼굴 탐지
- FR-005: 얼굴 영역 기준 품질 평가(밝기, 각도, 해상도)
- FR-006: 기준 미달 시 재촬영 유도

### 5.3 관상 결과 생성
- FR-007: 결과 카테고리 4종 제공
  - 성향
  - 대인관계
  - 일/커리어
  - 금전/행운 포인트
- FR-008: 결과는 긍정/중립 중심 톤으로 구성
- FR-009: 혐오, 차별, 공포 조장 표현 자동 필터링

### 5.4 결과 화면 및 공유
- FR-010: 결과 카드 이미지 자동 생성
- FR-011: SNS 공유(시스템 공유 시트 연동)
- FR-012: 결과 재생성(같은 사진 기준 variation 3회)

### 5.5 개인정보/보안
- FR-013: 동의 화면에서 처리 목적, 보관 기간, 삭제 방법 명시
- FR-014: 사용자 요청 시 원본/파생 데이터 삭제
- FR-015: 저장 데이터 암호화(전송 TLS, 저장 at-rest 암호화)

## 6) 비기능 요구사항

- NFR-001: 결과 생성 API 응답 시간 p95 3초 이내
- NFR-002: 앱 크래시 프리 세션 99.5% 이상
- NFR-003: 월간 가용성 99.9% 이상
- NFR-004: 접근성 기본 준수(대체 텍스트, 명확한 대비)
- NFR-005: 로그에 원본 얼굴 이미지를 저장하지 않음

## 7) 데이터 및 정책 요구사항

### 7.1 데이터 분류
- 민감 데이터: 얼굴 이미지(생체정보로 간주 가능)
- 일반 데이터: 기기 정보, 앱 버전, 이벤트 로그

### 7.2 수집 최소화
- 분석에 필요한 최소 이미지/메타데이터만 처리
- 디버깅 목적 원본 이미지 영구 저장 금지(옵트인 없는 경우)

### 7.3 보관/삭제
- 기본 정책: 분석 완료 후 원본 이미지 즉시 삭제(예: 24시간 내 자동 파기)
- 사용자가 저장 동의 시에만 보관
- 계정 삭제 또는 개별 삭제 요청 시 지체 없이 삭제

### 7.4 정책 고지
- 개인정보처리방침, 이용약관, 면책 고지 필수
- “본 결과는 오락용 해석 콘텐츠이며 사실/판단 근거가 아님” 문구 상시 노출

## 8) 기술 아키텍처(초안)

### 8.1 클라이언트
- 모바일 앱(React Native 또는 Flutter) 혹은 모바일 웹
- 카메라 촬영, 이미지 전처리(리사이즈/압축), 결과 렌더링

### 8.2 서버
- API Gateway
- Face Detection 서비스
- Interpretation Engine(룰/모델 기반)
- Content Safety Filter
- Result Card Renderer

### 8.3 저장소
- 단기 객체 저장소(암호화)
- 이벤트 분석용 로그 저장소(PII 제거)

## 9) UX/콘텐츠 가이드

- 과장/단정 표현 금지: “당신은 반드시 ~한다” 형태 금지
- 차별적 표현 금지: 성별/인종/외모 비하 금지
- 부정 예측 최소화: 공포/불행 단정 대신 개선 팁 제시
- 결과 템플릿은 사용자 불편을 줄이기 위해 중립적이고 존중하는 톤 유지

## 10) KPI 및 성공 지표

- D1 리텐션: 25% 이상
- 촬영 시작 대비 결과 도달율: 80% 이상
- 결과 공유율: 20% 이상
- 분석 실패율(미검출/저화질): 10% 이하
- 삭제 요청 처리 SLA: 24시간 이내 100%

## 12) 출시 계획(초안)

### 12.1 마일스톤
1. M1 (2주): UX 플로우/디자인, 동의/정책 문구 확정
2. M2 (3주): 촬영/업로드 + 얼굴 탐지 + 결과 API 연동
3. M3 (2주): 결과 카드/공유 + 로그/모니터링
4. M4 (1주): 보안 점검, 베타 테스트, 버그 수정

### 12.2 론칭 전략
- Closed Beta(100~300명) → 피드백 반영 → Open Beta
- 초기 타겟 채널: 숏폼/SNS 인플루언서 협업 및 공유 이벤트

## 13) 리스크 및 대응

- 리스크-001: 관상 결과에 대한 과신/오해
  - 대응: 오락용 고지 강화, 고위험 사용 금지 문구 고정
- 리스크-002: 얼굴 데이터 보안 이슈
  - 대응: 기본 미보관, 암호화, 접근통제, 삭제 자동화
- 리스크-003: 모델 편향
  - 대응: 다양한 데이터셋 검증, 정기 품질 점검, 신고 채널 운영
- 리스크-004: 법적 규제 변경
  - 대응: 분기별 법무 리뷰 및 약관/정책 업데이트

## 14) 오픈 이슈

- OI-001: 완전 온디바이스 처리 vs 서버 분석(비용/속도/보안 트레이드오프)
- OI-002: 비로그인 사용자 결과 이력 저장 정책
- OI-003: 글로벌 출시 시 국가별 생체정보 규제 대응 범위

## 15) 승인 기준

- 필수 기능(FR-001~FR-015) 구현 및 QA 통과
- 정책/약관/동의 UI 법무 검토 완료
- 보안 점검(Critical/High 취약점 0건)
- KPI 초기 가드레일(결과 도달율 70% 이상) 달성
